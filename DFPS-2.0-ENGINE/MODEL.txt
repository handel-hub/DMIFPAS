**The DAG definition contract — exactly what the dispatcher requires.**

The dispatcher consumes tasks as a flat list. 
It has no concept of a graph object. 
The graph structure lives entirely inside each task through `depends_on` and `children`.
 The DAG-BUILDER's job is to transform the MC's pipeline stage definitions into this flat task list with the correct field values.

---

**The MC sends this per job — what DAG-BUILDER receives:**

```json
{
  "job_id": "J1",
  "calculatedScore": 87.3,
  "pipeline": {
    "stages": [
      {
        "stage_id": "ingest",
        "plugin_id": "dicom-ingest-v2",
        "depends_on": []
      },
      {
        "stage_id": "segment",
        "plugin_id": "dicom-seg-v1",
        "depends_on": ["ingest"]
      },
      {
        "stage_id": "compress",
        "plugin_id": "compress-v3",
        "depends_on": ["segment"]
      }
    ]
  }
}
```

**The five rules DAG-BUILDER must enforce:**

**Rule 1 — ID format is `{job_id}_{stage_id}`.** The dispatcher uses a single flat registry.
 If two jobs both have a stage called `ingest`,their task IDs must differ. 
 Using `J1_ingest` and `J2_ingest` guarantees uniqueness. 
 The DAG-BUILDER generates these IDs — the MC only provides `stage_id` scoped within a job.

**Rule 2 — children is the computed inverse of depends_on.** The MC declares `depends_on` per stage.
 The DAG-BUILDER derives `children` by inverting this.
 For every stage S that lists stage P in its `depends_on`, P's children list must contain S's task ID.
  The dispatcher only decrements `_deps_remaining` via the `children` traversal — a missing child means a task is never unlocked.

**Rule 3 — pos_weight is computed from topological depth.** Entry tasks (no parents) get the highest weight. Leaf tasks (no children) get the lowest.
 A simple linear interpolation works:

```python
def compute_pos_weight(depth, max_depth):
    # depth 0 = entry, depth max_depth = leaf
    # weight range: 1.3 (entry) → 0.7 (leaf)
    return 1.3 - (0.6 * depth / max(max_depth, 1))
```

This means the dispatcher naturally prioritises unlocking the pipeline by doing entry tasks first, which is the correct behaviour for maximising throughput.

**Rule 4 — job_score is identical for all tasks from the same job.** The MC produces one `calculatedScore` per job.
 The DAG-BUILDER stamps this onto every task it creates from that job. 
 The dispatcher uses `job_score × pos_weight` as the numerator — if different tasks from the same job had different scores the ordering would be incoherent.

**Rule 5 — COSTING runs before DAG-BUILDER finalises tasks.
** The `duration_ms`, `cpu`, `ram`, and `spawn_latency_ms` fields must be populated by COSTING using the profile stores before the task list is handed to the dispatcher.
 DAG-BUILDER constructs the graph structure. COSTING fills the numeric cost fields. The dispatcher receives fully populated tasks.

---

**The two bugs that will silently corrupt scheduling if DAG-BUILDER gets them wrong:**

A missing child entry means a downstream task's `_deps_remaining` never reaches zero.
 That task sits in the blocked pool forever — it is never scheduled, never reported as failed, just silently dropped from the schedule.
  The Runtime Scheduler will wait for a completion signal that never arrives.

A task whose `cpu` or `ram` exceeds the safety-adjusted reservoir capacity will never pass `can_admit()`. 
The dispatcher has no escalation for this — it simply never schedules the task. The time jump loop will eventually exhaust the event heap with the task still in the 
ready pool, and the dispatcher will return a manifest that silently omits it. COSTING must validate resource feasibility before producing the task.
